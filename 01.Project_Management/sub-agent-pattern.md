# Sub-Agent Pattern ‚Äî CTO NDJSON Envelope Spec

> Resolves Audit Item A. PA always the face; sub-agent output never reaches user as raw text.

## Why

The naive approach ‚Äî let CTO emit free text and have PA make a second Claude call to "rephrase in PA voice" ‚Äî has three failures:
1. Doubles cost on every CTO interaction
2. Breaks streaming (user waits for CTO to finish before seeing anything)
3. Sentinel tokens (e.g. `__DONE__`) collide with code that legitimately contains them

This document specifies the alternative: a structured NDJSON envelope on stdout, parsed deterministically by PA, translated to PA voice via templated wrappers. LLM synthesis only when CTO explicitly requests it.

---

## Wire format

CTO writes one JSON object per line to stdout. Each line MUST be valid JSON, MUST end with `\n`, and MUST contain a `phase` key.

```json
{"phase": "plan",   "content": "I'll write monitor.ps1 that polls process X every 30s.", "needs_confirmation": true}
{"phase": "action", "content": "Writing sessions/abc/workspace/monitor.ps1"}
{"phase": "action", "content": "Running PowerShell syntax check on the new file"}
{"phase": "result", "content": "Wrote monitor.ps1 (847 bytes). Syntax check passed.", "files_changed": ["monitor.ps1"], "summary_needed": false}
```

### Phase types

| `phase` | Required keys | Optional keys | Meaning |
|---------|---------------|---------------|---------|
| `plan` | `content` | `needs_confirmation` (bool, default false) | What CTO is about to do. If `needs_confirmation=true`, PA creates an escalation and waits. |
| `action` | `content` | ‚Äî | Live progress line. Streamed to user. |
| `result` | `content` | `files_changed` (list[str]), `summary_needed` (bool, default false) | Final outcome. If `summary_needed=true`, PA makes one Claude call to synthesise. |
| `error` | `content`, `code` | `retriable` (bool) | Failure. PA creates escalation `(a) retry (b) abort`. |
| `ask` | `content`, `options` (dict) | ‚Äî | CTO needs user input mid-task. PA creates escalation with the given options. |

### Stderr

Anything CTO writes to stderr is captured to `sessions/{id}/cto.err.log` and never surfaced to the user. Used for warnings, debug output, claude CLI internal messages.

### Non-JSON stdout

If CTO accidentally writes a non-envelope line to stdout (e.g. raw print statements from inside generated code), PA logs it to `sessions/{id}/cto.stray.log` and discards. Does not crash or surface to user. CTO's CLAUDE.md instructs against this, but the parser is defensive.

## PA-side wrappers (templated, deterministic)

`orchestrator/proxy/adapters/wrapper_templates.py`:

```python
TEMPLATES = {
    "plan_confirm": "I want to {content}\n\nProceed?\n  (a) yes\n  (b) cancel",
    "plan_no_confirm": "Working on it: {content}",
    "action": "  ‚Üí {content}",
    "result_simple": "Done. {content}" + (" Files: {files_changed}" if files_changed else ""),
    "error": "Hit an error: {content} ({code})\n  (a) retry\n  (b) abort",
}
```

PA's wrapper layer is a pure function: `(envelope, context) ‚Üí user_facing_text`. Zero LLM calls.

**The only place a Claude call is justified** is when the CTO sets `result.summary_needed=true` ‚Äî typically for multi-file refactors or research summaries where templated output would be unhelpful. CTO is instructed to set this sparingly.

## Spawner-written CLAUDE.md (CTO instructions)

When `spawner.spawn()` runs, it writes `sessions/{id}/.claude/CLAUDE.md` containing:

```markdown
# CTO Sub-Agent Instructions

You are the CTO sub-agent for a Personal Assistant system. The user is talking
to the PA, who routes coding/file/script tasks to you.

## Output protocol ‚Äî STRICT
Every line you write to stdout MUST be a single-line JSON object with a `phase` key.

Allowed phases:
- "plan"   ‚Äî describe what you're about to do. Set `needs_confirmation: true`
            for any file write, script execution, or external action.
- "action" ‚Äî short line on what's happening right now (one per major step).
- "result" ‚Äî final outcome. Set `summary_needed: true` ONLY for multi-file
            results or analysis output where a templated wrap-up would lose meaning.
- "error"  ‚Äî failure. Include `code` and `retriable`.
- "ask"    ‚Äî request for user input mid-task. Include `options` dict.

DO NOT write free text on stdout. Use stderr for debug output (it's logged but
not shown to user).

## Workspace
You may read and write files only inside: {{workspace_path}}
Maximum file size: 10 MB. Maximum total workspace size: 500 MB.

## Available adapters (callable via your tools)
- file_read, file_write, brave_search

## Task brief
{{task_brief ‚Äî generated by PA from session context}}

## Context refresh protocol
PA may send messages prefixed with `[brief-update]`. These are NOT new requests
‚Äî treat as background context for whatever you're already doing.

## Idle / completion
When you finish a request, emit `result` and wait. Do not exit. PA's spawner
will reap you on idle (15 min) or on `@PA` switch.
```

Static sections are constants. Only the `task_brief` and `workspace_path` vary per spawn.

## Lifecycle

```
@CTO <request>
    ‚îÇ
    ‚ñº
PA: is there a live CTO subprocess for this session?
    NO  ‚Üí spawner.spawn(session_id, brief_context):
              - Generate brief via ONE Claude call
              - Write CLAUDE.md
              - Popen claude.exe (CREATE_NEW_PROCESS_GROUP)
              - Register PID in sessions.cc_pid
    YES ‚Üí reuse
    ‚îÇ
    ‚ñº
PA writes <request> to CTO stdin
    ‚îÇ
    ‚ñº
PA reads CTO stdout line-by-line, parses each as envelope JSON
    ‚îÇ
    ‚ñº
For each envelope:
    - "plan" + needs_confirmation ‚Üí create escalation, wait
    - "action" ‚Üí stream wrapper line to user
    - "result" + simple ‚Üí templated done message
    - "result" + summary_needed ‚Üí ONE Claude synthesis call
    - "error" ‚Üí create escalation
    - "ask" ‚Üí create escalation with custom options
```

## Cost analysis

**Common case** (write a file, run a script, fix a bug): zero extra LLM calls beyond the CTO's own internal usage. PA's wrapping is pure templating.

**Spawn case** (first @CTO of a session): one Claude call for the task brief. Amortised across the entire CTO session (which may be many turns).

**Synthesis case** (`summary_needed=true`): one Claude call. Only when CTO determines templated output is insufficient. Expected frequency: <20% of CTO results.

**Compared to v2 plan** (synthesis on every result): 5x cost reduction in typical use.
