# Job System — Definition + Executor

> Resolves Audit Item B. Phase 1.2 feature. Hybrid: human-readable + machine-readable.

## Three job types

### Option A — Recurring (file + cron)
Plain-English description plus a machine-generated YAML execution plan. File at `jobs/{name}.md`. Scheduled via APScheduler. Deterministic at runtime.

### Option B — One-off
PA chooses adapters and executes immediately based on the user's current request. No file. Cost recorded in `cost_ledger`.

### Option C — Interest profile
`config/interests.md`. Free-form English. PA reads as context for any research-type job. Updated when user says "remember I'm interested in X."

---

## Option A — Job file format

```markdown
# jobs/<name>.md

## What I want
<plain English — the user-editable source of truth>

## Execution Plan
<!--
Generated by PA. Do not edit by hand.
To regenerate after changing "What I want", say: @rebuild-plan jobs/<name>.md
-->
```yaml
version: 1
trigger:
  cron: "<cron expression>"
  timezone: "<IANA tz>"
steps:
  - id: <unique step id>
    adapter: <adapter name from registry>
    params: { ... }            # validated against adapter manifest
    on_error: escalate         # escalate | skip | abort (default: escalate)
  - id: ...
    adapter: ...
    params:
      input_from: "$<step_id>.data"   # interpolation from previous step
```

## Last Run
<auto-updated by job_runner after each execution: timestamp + status + cost>
```

### Example

```markdown
# jobs/hn-daily.md

## What I want
Scrape the top 10 stories from Hacker News every morning at 8am Paris time
and email me a digest at christophorous@gmail.com.

## Execution Plan
<!-- Generated by PA. Do not edit. To regenerate: @rebuild-plan jobs/hn-daily.md -->
```yaml
version: 1
trigger:
  cron: "0 8 * * *"
  timezone: "Europe/Paris"
steps:
  - id: fetch
    adapter: playwright_web
    params:
      operation: extract_links_top_n
      url: "https://news.ycombinator.com"
      n: 10
      selector: "tr.athing"

  - id: render
    adapter: template_render
    params:
      template: "hn_digest.md.j2"
      context_from: "$fetch.data"

  - id: deliver
    adapter: email_send
    params:
      to: "christophorous@gmail.com"
      subject: "HN Daily — {{ today }}"
      body_from: "$render.data"
      content_type: "text/markdown"
```

## Last Run
2026-05-08 08:00:03 — ok — 0.0000 USD (0 LLM calls; deterministic execution)
```

## Plan validation

Adapter Manifest Registry: every adapter exports a `manifest` describing operations + required/allowed params with types.

```python
# Example adapter manifest
PlaywrightWebManifest = AdapterManifest(
    adapter_name="playwright_web",
    operations={
        "fetch_url": {
            "required": {"url": str},
            "allowed":  {"timeout_s": int},
            "returns": "html: str",
        },
        "extract_links_top_n": {
            "required": {"url": str, "n": int},
            "allowed":  {"selector": str, "domain": str},
            "returns": "data: list[{title, url}]",
        },
    },
    allowed_callers={Caller.PA, Caller.CTO_SUBAGENT, Caller.JOB_RUNNER},
)
```

Validator (`job_runner.validate_plan`):
1. Top-level YAML schema: `version`, `trigger`, `steps`
2. Each step: `id` unique, `adapter` exists in registry, `adapter.manifest` allows `JOB_RUNNER`
3. `params` matches the manifest's required/allowed for the chosen operation
4. `input_from` interpolations reference step ids that appear earlier in the list

Any failure → `Result(ok=False, error=BAD_INPUT)` and the file is NOT written. PA reports the validation error to the user.

## Plan generation (one-time, at job creation)

PA's plan-author flow:

1. User describes a workflow in chat
2. PA confirms ambiguous bits (timing, recipients)
3. PA calls Claude API with system prompt extension:
   ```
   You are converting a user's workflow description into a structured Execution Plan.
   Available adapters and their operations:
   <adapter manifest registry as a JSON block>

   Output format: a single YAML object matching this schema: <schema>
   ```
4. PA validates the returned YAML against the registry
5. If invalid → ask Claude to fix with the validation error → max 2 retries → escalate to user if still bad
6. If valid → write `jobs/{name}.md` with both blocks via FileWriteAdapter
7. Insert `jobs` row with `plan_checksum = sha256(what_i_want_block)`

## Plan invalidation (on user edit)

Every job_runner.run() starts with:
```python
content = read_file(job.file_path)
what_i_want = parse_section(content, "## What I want")
current_checksum = sha256(what_i_want)
if current_checksum != job.plan_checksum:
    create_escalation(
        session_id=job.created_by_session_id,
        options={"a": "regenerate plan now", "b": "run with old plan", "c": "skip this run"},
        context={"job_id": job.id, "checksum_mismatch": True},
    )
    return  # halt execution; user must respond
```

User responses:
- `a` → trigger PA's `@rebuild-plan` flow (one Claude call) → update `plan_checksum` → re-trigger run
- `b` → execute with the existing plan
- `c` → skip; mark in `job_runs` as status=`skipped`

## Runtime execution (deterministic)

`orchestrator/job_runner.py`:

```python
async def run(job_id: str):
    job = await store.get_job(job_id)
    content = read_file(job.file_path)
    plan_yaml = parse_section(content, "## Execution Plan")  # extracts the ```yaml...``` block
    plan = yaml.safe_load(plan_yaml)

    # Checksum verification
    if sha256(parse_section(content, "## What I want")) != job.plan_checksum:
        await escalate_checksum_mismatch(job)
        return

    # Validation (defensive — already validated at write time)
    validate_plan(plan)

    # Execute
    job_run_id = await store.create_job_run(job.id)
    step_results = {}
    total_cost = 0.0
    try:
        for step in plan["steps"]:
            params = interpolate(step["params"], step_results)  # resolves $foo.data
            intent = Intent(
                kind=adapter_kind_for(step["adapter"]),
                payload={"operation": params.get("operation"), **params},
                session_id=job.id,            # job-scoped session
                mode="PA",
                caller=Caller.JOB_RUNNER,
                deadline_s=adapter_default_timeout(step["adapter"]),
                attempt=0,
            )
            result = await dispatcher.dispatch(intent)
            total_cost += result.cost_usd
            if not result.ok:
                if step.get("on_error", "escalate") == "skip":
                    step_results[step["id"]] = None
                    continue
                elif step.get("on_error") == "abort":
                    raise StepFailed(step["id"], result.error)
                else:
                    await escalate_step_failure(job, step, result.error)
                    return
            step_results[step["id"]] = result.data

        await store.complete_job_run(job_run_id, status="ok",
                                     summary=summarise_steps(step_results),
                                     cost_usd=total_cost)
        await store.insert_event(
            session_id=job.created_by_session_id,
            channel="telegram",  # or whatever the user prefers
            kind="job_complete",
            payload={"job_name": job.name, "run_id": job_run_id, "summary": "..."}
        )
    except StepFailed as e:
        await store.complete_job_run(job_run_id, status="error",
                                     summary=str(e), cost_usd=total_cost)
        await store.insert_event(... kind="job_complete", payload={..., "error": True})
```

## Notification flow

Job runner inserts row into `events` table on completion. The FastAPI process's `events_consumer` (running in a separate Python process) picks it up within 500ms and dispatches to:
- Active WebSocket for the user (web channel)
- Telegram via `bot.send_message` using `sessions.telegram_chat_id` (telegram channel)

Long results (>4000 chars on Telegram) are delivered as a `.md` document attachment.

## Cost characterisation

| Phase | Claude calls |
|-------|--------------|
| Job creation (one-time) | 1 (plan generation) + maybe 1-2 retries on invalid YAML |
| `@rebuild-plan` (one-time per edit) | 1 |
| Scheduled run (recurring) | 0 |
| User notification | 0 (templated) |

Compared to "LLM call per fire" plan: cost is amortised over the lifetime of the job. A daily job that runs for a year incurs ~1 LLM call total instead of ~365.

## Edge cases

- **Site changes structure** (e.g. HN HTML changes): `playwright_web.extract_links_top_n` returns empty list. Step succeeds with no data. Downstream `template_render` produces an empty digest. User notices and asks PA to fix.
- **Adapter failure** (e.g. Brave Search down): retried per guardrails; on terminal failure, escalation per `on_error` policy.
- **Plan execution exceeds budget**: pre-step cost check — if `total_cost + step.estimated_max > $5/day`, halt and escalate.
- **User edits Execution Plan directly** (not advised): plan_checksum is on `## What I want` only, so direct edits to the YAML block are not detected. Documented in the file's HTML comment as "do not edit by hand."
- **Adapter manifest changes** (e.g. new required param added): plan validation at execution time catches missing required params; escalation triggered.

---

## What this design rules out

- Non-deterministic execution (LLM call per run) — REJECTED in audit
- Hand-written YAML by user — supported but not recommended (no syntax help)
- Single-block YAML-only job files — loses the "describe in English" UX
